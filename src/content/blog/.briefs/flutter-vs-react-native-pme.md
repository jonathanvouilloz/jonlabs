## Structure cible (H2/H3)

### Introduction
Quand un client PME me demande quelle techno choisir pour son app, je lui réponds toujours : "Ce qui importe pour toi, c'est le résultat — pas le framework." Mais la techno influence le coût et le délai. Voici ce que tu dois savoir.

### H2 : La différence en 30 secondes
- **Flutter** (Google, 2018) : compile en natif, une codebase pour iOS + Android + Web. Performance proche du natif. Langage : Dart.
- **React Native** (Meta, 2015) : JavaScript/TypeScript, pont vers les composants natifs. Grande communauté, plus de libraries tierces.
- Les deux sont cross-platform : une seule base de code pour iOS et Android = moins cher que 2 apps natives.

### H2 : Tableau comparatif Flutter vs React Native

| Critère | Flutter | React Native |
|---|---|---|
| Performance perçue | Excellente (rendu propre) | Bonne (quelques jank possibles) |
| Vitesse de développement | Rapide une fois maîtrisé | Rapide (surtout si équipe JS) |
| Maintenance | Simple (une codebase) | Dépend des libs tierces |
| Taille communauté | Grande (en forte croissance) | Très grande (plus mature) |
| Ecosystem libraries | Grandissant | Très riche |
| Coût développement | Comparable | Comparable |
| Support Google/Meta | Fort (Google) | Modéré (Meta moins impliqué qu'avant) |
| Maturité | 7 ans, stable | 10 ans, très mature |

### H2 : Pour quel projet je choisis Flutter
- App B2B avec UI complexe et animations
- Projet qui veut aussi une version Web (Flutter Web)
- Client qui n'a pas de devs JS en interne (équipe Dart dédiée)
- App avec besoin de performance graphique élevée
- Recommandation Jon Labs : Flutter est mon choix par défaut depuis 2024

### H2 : Pour quel projet je choisis React Native
- Startup avec équipe JavaScript existante (réutilise les compétences)
- Besoin d'un écosystème de libraries très riche immédiatement
- Intégration de libs tierces spécialisées (pas encore dispo en Flutter)
- Projet avec fort composant web (partage de code React ↔ RN)

### H2 : Et les autres options ?

**Expo (sur React Native)** : idéal pour prototyper rapidement ou MVP. Simplifie la config. Limite : certaines features natives nécessitent "eject".

**PWA** : si ton app n'a pas besoin d'être sur les stores et que les users l'utilisent via navigateur — souvent sous-estimée. Voir article dédié.

**App native pure (Swift + Kotlin)** : uniquement si tes besoins sont très poussés (AR, jeux, hardware très spécifique) ou si les performances sont critiques. 2× plus cher.

### H2 : La recommandation Jon Labs

Flutter. Pour une PME romande qui commande sa première app, Flutter offre le meilleur rapport performance/coût/maintenance en 2026. React Native reste excellent — mais Flutter a pris une longueur d'avance côté stabilité et rendu visuel. Et l'avantage d'une codebase unique est réel.

Si tu as déjà une équipe React Native en interne : reste sur RN. Sinon, pars sur Flutter.
